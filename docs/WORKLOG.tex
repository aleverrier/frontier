\subsection*{2026-06-15 BB144 Reproduction Success Criteria}

\begin{itemize}
\item Added a README section explaining how a fresh user can tell that the BB144/Gross \(p=0.001\), \(\Delta=12\), \(K=512\), \texttt{\detokenize{fwd_bwd_committee}} reproduction worked.
\item Documented the expected \texttt{\detokenize{frontier-dem-info}} matrix output: \texttt{\detokenize{memory_X}} and \texttt{\detokenize{memory_Z}} detector matrices are \texttt{\detokenize{936x8784}}, logical matrices are \texttt{\detokenize{12x8784}}, with 12 noisy rounds and \texttt{\detokenize{deadline_reorder}}.
\item Documented the expected generated workload shape: 20,000 side rows for 10,000 paired full-frame shots, 40 replay shard tasks when \texttt{\detokenize{--shards-per-side 20}} is used, and a complete native replay recorded in \texttt{\detokenize{run_metadata.json}}.
\item Clarified that \texttt{\detokenize{summary_by_scope.csv}} contains \texttt{\detokenize{memory_X}}, \texttt{\detokenize{memory_Z}}, and \texttt{\detokenize{combined}}, and that the \texttt{\detokenize{combined}} row is the strict full logical FER over paired side rows.
\item Avoided claiming \(\mathrm{FER}=0\): at \(p=0.001\), 10k shots is a smoke-scale reproducibility sample, and no observed failures should be reported as below the resolution of that sample rather than as evidence that the FER is zero.
\item Avoided quoting a wall-clock target in the README. Timing is machine-dependent; the README now asks timing reports to include machine/Python/workers/native availability/batch size and points to transition-evaluation counts for more machine-independent comparison.
\end{itemize}
